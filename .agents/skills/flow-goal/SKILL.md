# Flow Goal

本 skill 只生成可复制的 goal prompt，用于把想法变成可执行目标；默认不创建 goal、不开始执行。

## 触发

- 用户想生成一段可直接用于创建 goal 的提示词。
- 用户给出模糊想法，需要整理成目标、范围、完成定义和验证规则。
- 用户要启动长期任务，但还不想直接执行。
- 用户要求“一键生成 goal prompt”或“让主 agent 写 goal 提示词”。

## 边界

- 默认只生成 goal prompt，不创建 goal，不开始执行。
- 只有用户明确说“创建并开始”“直接创建 goal”时，才进入 goal 创建或执行流程。
- 不把具体运行后端、价格、上下文大小或供应商能力写进 prompt。
- 不把不确定事实写成确定要求；需要当前事实时，在 prompt 中要求先读 `docs/websearch/index.md` 并联网核验。
- 输出面向复制使用，避免解释性长文。

## 生成流程

1. 先读用户原始意图，提取：
   - `goal`
   - `background`
   - `in_scope`
   - `out_of_scope`
   - `constraints`
   - `definition_of_done`
   - `verification`
   - `output_format`
2. 如当前仓库会影响目标，读取 `AGENTS.md` 和必要目录结构。
3. 信息缺失但不影响生成时，用 `<待补充>` 占位；缺失会导致高风险误执行时，先问用户。
4. 判断是否需要：
   - `$flow-kickoff`
   - `$flow-research`
   - 契约索引
   - Sprint 契约
   - Agent 边界矩阵
   - 评估计划
5. 生成一段可直接复制的 prompt。

## Prompt 结构

生成的 goal prompt 应包含：

- 目标：一句话说明最终要完成什么。
- 背景：仓库、当前状态、已有约束、已知资料。
- 范围内：必须完成的事项。
- 范围外：明确不做的事项。
- 完成定义：可观察、可验证的完成条件。
- 执行规则：文档契约、代码范围、联网、权限、安全、停止条件。
- 验证要求：命令、人工检查、证据、失败阈值。
- 输出格式：最终回复必须包含的字段。

## 质量规则

- 目标必须具体，不能只写“优化”“完善”“处理一下”。
- 范围内和范围外都要写，避免任务膨胀。
- 完成定义必须可验证。
- 复杂任务必须要求先规划，再执行。
- 会改变需求、架构、公开 API、配置、数据、权限、工作流或验收时，要求先更新契约索引或 Sprint 契约。
- 有联网需求时，要求先读 `docs/websearch/index.md`，再联网。
- 有破坏性、权限扩大、部署、迁移、凭据或外部可见操作时，要求先确认。
- 长任务必须有停止条件，避免无界搜索、无界重试或过度验证。

## 输出

只输出以下两段：

1. `Goal Prompt`
2. `缺失信息`

`Goal Prompt` 使用可复制代码块。

`缺失信息` 只列会影响执行质量的问题；没有则写“无”。
